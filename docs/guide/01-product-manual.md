# 产品说明书 — Workflow Studio

## 一句话
**Workflow Studio 是一个 Serverless Easy API Maker：** 你在可视化画布上把"消息丰富化 + 条件判断 + 多通道下发"编排成一个工作流，发布之后它就是一个**可调用的 API**——不写一行后端代码。

## 它解决什么问题
集成场景里，经常要做这样一类后端：收到一条交易/业务消息 → 调几个外部接口补全数据 → 按条件判断走不同分支 → 把结果下发到短信/邮件等通道，并留下可追溯的运行记录。手写这套后端，意味着搭服务、写路由、接数据库、做幂等和重试、上线运维。

Workflow Studio 把这套"消息处理后端"变成**画布上的编排 + 一键发布**：节点表达步骤，连线表达顺序，JSONPath 规则表达条件，平台负责执行、幂等、重试和记录。

## 为谁
| 角色 | 用它做什么 |
|---|---|
| 集成工程师 | 编排工作流、配置应用、发布 API |
| 应用开发者 | 调用已发布的 API（提交运行请求） |
| 运维 / QA | 在画布上试跑、在运行记录里排查 |
| 产品 / 业务 | 用 AI Explain 读懂一个工作流在做什么 |

## 凭什么（对比）
| 方案 | 取舍 |
|---|---|
| **手写后端** | 最灵活，但要自己搭服务/数据库/幂等/重试/运维，慢且重 |
| **Lambda + API Gateway** | 仍要写代码、配网关、管部署；逻辑散在代码里不可视 |
| **Zapier / n8n 等** | 偏"触发器→动作"自动化；本平台聚焦"消息丰富化 + 条件下发 + 可追溯记录"，且产物是**你自己的 API** |
| **Workflow Studio** | 可视化编排、JSONPath 规则、内置幂等/重试/记录、一键发布为 API；代价是节点类型与表达能力目前较聚焦（见"能力边界"） |

## 今天能做什么
- 可视化画布编排（拖放节点、连线、配置）
- 6 种节点：HTTP 丰富化、条件分支、消息下发、函数步骤（见[节点参考](reference/nodes.md)）
- JSONPath 规则：满足条件才执行某步
- 同步 / 异步执行，按应用配置
- **幂等**（同一业务关联键不重复处理）与**应用级重试**
- **运行记录**：每次调用可查、可下钻父子/重试链
- **配置历史与回滚**（基于修订）
- **一键发布（Deploy）** 到目标环境
- **AI 辅助**：Explain（解释工作流）、Generate（自然语言生成工作流）
- **JSON 导入 / 导出**：把工作流当作可移植的"源码"

## 能力边界（诚实说明，避免踩坑）
> 这些是**当前**的限制；部分在路线图上（见 [`../TODO-doc-gaps.md`](../TODO-doc-gaps.md)）。
- **调用响应目前是"成功/失败"而非"计算结果"**：发布的 API 成功时返回 `200` 空体，结果通过**运行记录**查看，而不是直接返回给调用方的 JSON 响应体。"由工作流塑造同步响应"在路线图上。
- **统一入口**：所有应用共用 `POST /api/workflow?applicationName=...`，暂无"每个 API 自定义路径/方法"。
- **发布 API 的入站鉴权 / 限流**：暂未内置（路线图）。
- **控制流**：条件为二元分支（IFELSE）；暂无循环 / switch / 子流程节点。
- **规则**：每条规则是单个 JSONPath；暂无通用表达式/转换语言。

## 关键术语速览
应用（Application）= 一个工作流的容器与发布单元；工作流（Workflow）= 节点 + 连线；节点（Node）= 一个步骤；规则（Rule）= JSONPath 守卫；丰富化（Enrichment）/ 下发（Dispatch）= 执行的两个阶段；运行记录（Record）= 一次调用的可追溯结果。详见[核心概念](03-concepts.md)。
