# 核心概念与心智模型

读完这页，你就懂了 Workflow Studio 的"语言"。

## 心智模型一图
```
一条消息(JSON)  ──▶  [应用 = 你的 API]
                         │
              ┌──────────┴───────────┐
              │  丰富化阶段(Enrichment) │  按连线顺序，规则命中才执行；
              │  调外部接口、合并数据   │  结果不断 merge 进"运行负载"
              └──────────┬───────────┘
                         ▼
              ┌──────────────────────┐
              │  下发阶段(Dispatch)    │  多通道下发，写入通道状态
              └──────────┬───────────┘
                         ▼
                 [运行记录 Record]  ← 可查、可下钻、可追溯
```

## 词汇表

| 术语 | 含义 |
|---|---|
| **应用 Application** | 一个工作流的容器与发布单元，有唯一 `applicationName`。**它就是你的 API 的标识。** |
| **工作流 Workflow** | 节点 + 连线构成的有向图，决定执行逻辑与顺序。可导入/导出为 JSON。 |
| **节点 Node / 插件 Plugin** | 一个步骤。共 6 种类型（见[节点参考](reference/nodes.md)）。 |
| **连线 Edge（uiMap）** | 节点间的有向连接，决定执行顺序；IFELSE 用 `*_true` / `*_false` 表达分支。 |
| **规则 Rule** | 挂在节点上的**单个 JSONPath**；对当前运行负载**全部匹配**才执行该步，否则跳过（无隐式默认分支）。 |
| **运行负载 Runtime Payload** | 一条贯穿执行的可变 JSON：入站 body + 各步丰富化结果不断 merge。 |
| **丰富化 Enrichment** | 派发前的阶段：调外部接口、按规则合并数据。 |
| **下发 Dispatch** | 丰富化之后的阶段：向通道（短信/邮件等）下发，写入各通道状态与总状态。 |
| **同步 / 异步** | 按应用配置：同步=处理完才返回；异步=立即返回、后台处理。 |
| **幂等 Idempotency** | 同一应用下同一**业务关联键**（`X-Request-Correlation-Id`）只处理一次，重复请求被拒（`M0002`）。 |
| **重试 Retry** | 命中应用级重试策略时产生可关联的重试记录，次数可追溯。 |
| **运行记录 Record** | 一次调用的可追溯结果：每步状态、通道状态、父子/重试链。 |
| **发布 Deploy** | 把应用 + 工作流推送到目标环境，使其可被调用。 |
| **Explain / Generate** | AI 辅助：解释一个工作流 / 用自然语言生成工作流。 |

## 三个最容易混的点
1. **"应用"就是"API"**：建应用 = 开一个 API；`applicationName` = API 标识。
2. **规则是"守卫"不是"赋值"**：JSONPath 命中→执行该步；不命中→跳过。它判断"要不要做这步"，不做数据转换。
3. **结果在记录里，不在响应里**：当前调用成功只回 `200` 空体，执行结果去[运行记录](#)看（"由工作流塑造同步响应"在路线图上）。

## 一个工作流长什么样（JSON）
```json
{
  "pluginList": [ { "id": 1, "ruleList": [{"key":"$.customerId"}], "action": {"type":"CONSUMER", ...} } ],
  "uiMapList": [ { "id": "e1", "source": "1", "target": "2" } ]
}
```
完整 schema 见[工作流 JSON 参考](reference/workflow-json.md)。
